# 032-HermesGo 大文件与发布策略规划

## 元信息
- 编号：32
- 日期：2026-04-19
- 参考：`.cursorrules`、`create_hermes_go/README.md`、`create_hermes_go/Create-HermesGo.ps1`、`HermesGo/runtime/hermes-agent/web`、`HermesGo/runtime/hermes-agent/hermes_cli`
- 目的：把 HermesGo 的源码、脚本、运行时大文件、离线包和发布资产分层，避免把所有 zip / exe / pyd / dll 直接塞进 git 历史。

## 问题背景

当前仓库里同时存在：
- 源码与脚本
- 构建产物
- 便携运行时
- Ollama / Python / Windows 依赖大文件
- 输出包和缓存 zip

如果这些内容全部直接进 git，会带来：
- 仓库体积持续增长
- clone / pull 很慢
- review 很难看清真实代码变更
- 版本回滚和分支协作成本高
- CI / 自动化打包容易被无关二进制噪音干扰

## 目标

1. 让用户仍然可以“解压即用”。
2. 让源码仓库保持轻量，便于日常开发和审查。
3. 让大包有稳定的下载入口和校验方式。
4. 让 HermesGo 的本地运行时和用户当前系统登录彼此隔离。

## 建议方案

### 1. git 只提交源代码和发布脚本

建议进入 git 的内容：
- `HermesGo/runtime/hermes-agent/hermes_cli/*.py`
- `HermesGo/runtime/hermes-agent/web/src/*`
- `HermesGo/tools/*.ps1`
- `create_hermes_go/*.ps1`
- `create_hermes_go/*.md`
- `docs/*.md`
- 小体积配置样例、清单、校验文件

不建议进入 git 的内容：
- `runtime/python311/`
- `runtime/ollama/`
- `runtime/hermes-agent/web_dist/`
- `*.zip`
- `*.exe`
- `*.dll`
- `*.pyd`
- `*.whl`
- `node_modules/`
- 构建缓存和临时输出

### 2. 离线包独立发布

把“完整可运行包”作为独立产物发布，而不是依赖源码仓库本身：
- GitHub Releases
- 百度网盘或其他网盘
- 内部文件服务器

离线包建议包含：
- `HermesGo.exe`
- `runtime/`
- `home/`
- `installers/`
- `web_dist/`
- 需要用户直接运行的启动脚本

### 3. 首次启动下载机制

如果某些包不想打进离线包，建议改成首次启动下载：
- 启动脚本先检测本地文件是否存在
- 不存在时按清单下载
- 下载后做 hash 校验
- 失败时给出明确错误和手动下载路径

适合首次下载的内容：
- 大模型
- 可选 GPU 后端
- 某些平台依赖安装器
- 不常变的辅助工具包

### 4. 用清单描述“大文件”

建议加一个统一清单，例如：
- `assets.manifest.json`
- `release-manifest.json`
- `checksums.txt`

清单至少记录：
- 文件名
- 大小
- SHA256
- 下载源
- 是否必需
- 是否可按需下载

### 5. 保留一键打包脚本

继续保留并强化：
- `create_hermes_go/Create-HermesGo.ps1`
- `create_hermes_go/Bundle-Ollama.ps1`

脚本职责：
- 从源码构建完整离线包
- 把大文件复制到发布目录
- 生成校验文件
- 必要时上传到网盘或发布页

## 推荐落地顺序

1. 先补 `.gitignore` 和发布清单，明确哪些文件不进 git。
2. 再把 `create_hermes_go/output/HermesGo` 定义成纯产物目录。
3. 然后把离线包上传到网盘或 Releases。
4. 最后加首次启动下载和 hash 校验。

## 风险

- 如果把全部大文件直接保留在源码仓库，后续每次打包都会越来越慢。
- 如果只依赖网盘，没有校验文件，用户很容易拿到损坏包。
- 如果下载逻辑没有清单，后续补文件会很乱。

## 结论

建议采用“源码进 git，运行时大包外置，离线包单独发布，下载有清单和校验”的模式。
这比“全量提交所有 zip 和运行时二进制”更适合长期维护。
