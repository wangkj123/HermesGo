**说明**：本方案基于官方公开资料和当前仓库已有文档。当前结论是 Hermes Windows 侧更现实的入口是 Pinokio；因此 HermesGo 采用“本地启动入口 + Hermes 官方 Pinokio 页面”的方式实现，不依赖 WSL2，也不承担安装职责。

## 1. 结论

1.1 HermesGo 已建立为独立目录，不再把入口文件散放在仓库根目录。

1.2 启动器采用 Windows batch，实现双击 `HermesGo.bat` 即可进入入口逻辑。

1.3 启动器优先检测本机 Pinokio；如果已安装，先启动 Pinokio，再打开 Hermes 官方 Pinokio 页面。

1.4 如果本机没有 Pinokio，启动器直接打开 Hermes 官方 Pinokio 页面。

1.5 这套流程不负责安装，也不依赖 WSL2。

## 2. 目录结构

2.1 `HermesGo/HermesGo.bat`

2.1.1 Windows 一键入口。

2.2 `HermesGo/logs/`

2.2.1 保存启动日志。

## 3. 启动逻辑

3.1 第一优先级：本机 Pinokio。

3.2 第二优先级：Hermes 官方 Pinokio 页面。

3.3 启动器不做安装，不提供 `-DryRun` 之类的辅助模式，保持批处理脚本的直接性。

## 4. 验证结果

4.1 已检查 `HermesGo.bat` 的目录逻辑，确认入口只保留在 `HermesGo/` 内。

4.2 已验证无 Pinokio 时会直接打开 Hermes 官方 Pinokio 页面。

4.3 已验证有 Pinokio 时会先尝试启动 Pinokio，再打开 Hermes 官方 Pinokio 页面。

4.4 `HermesGo/logs/HermesGo.log` 会在运行时自动生成，属于目录内日志文件。

## 5. 当前限制

5.1 Hermes 官方公开资料给出的 Windows 路径是 Pinokio，不是原生 Windows 直装。

5.2 这意味着 HermesGo 目前是“Windows 便携启动入口”，不是把 Hermes 重写成 Windows 原生程序。

5.3 如果后续要做真正的离线绿色包，还需要另外评估 Pinokio 和 Hermes 的离线安装链路。

## 6. 后续建议

6.1 如果你后面要再做真正的“拷贝即用”，再单独规划离线依赖，不要混进这个启动器。

6.2 如果你要继续强化绿色版，可以再加一个本地模型目录、离线缓存目录和更细的启动检测。

6.3 如果你要真正做成“拷贝即用”，下一步应该先固定 Pinokio 包和 Hermes 版本，再做一次完整离线验证。
