# HermesGo 先降门槛后做技能共享的路线判断

**目的**：记录当前路线判断：先把 HermesGo 做成小白可用、低安装门槛、低理解成本的入口，再叠加 skill 共享、讨论、评分与互助机制。

## 1. 路线结论

1.1 当前优先级应当是：

- 先把 HermesGo 做成“能装、能开、能跑、能自检”的入口
- 再做 skill 共享、评分、互助、社区化分发

1.2 原因很直接：

- 安装门槛不降下来，后面的共享机制没有使用基础
- 使用入口不顺，社区再热闹也进不来
- 对小白用户来说，第一阻碍不是“有没有技能市场”，而是“能不能顺利启动”

1.3 因此，技能共享不是当前主矛盾，HermesGo 的可用性才是主矛盾。

## 2. 这条路线为什么合理

2.1 Hermes 已经具备一定的技能体系和可扩展性，但这不等于普通用户能直接用起来。

2.2 真正决定采用率的，是入口复杂度：

- 是否需要理解多个概念
- 是否需要手工装依赖
- 是否需要反复配置环境
- 是否需要读很多说明才能开始

2.3 所以当前阶段最重要的不是扩张功能面，而是把使用链路压短。

## 3. 当前应该做什么

3.1 HermesGo 的目标应当继续收敛为：

- 一键启动
- 自动检测依赖
- 缺失时给出明确下一步
- 自检可见
- 报错可定位
- 默认配置能直接跑

3.2 这类工作优先级高于：

- 技能共享市场
- 评分机制
- 讨论论坛
- 激励分配

3.3 因为后者都建立在“用户已经愿意用”的前提上。

## 4. 为什么不该现在就做共享层

4.1 共享层当然有价值，但它属于第二阶段。

4.2 如果过早上共享层，常见问题会是：

- 用户还没装好，就被功能淹没
- 维护者先被治理问题拖住
- 评分和激励机制比核心产品更复杂
- 项目看起来很大，实际上可用性不高

4.3 所以当前应该避免“先做生态，再做入口”的倒序。

## 5. 与现有路线的关系

5.1 这条判断和前面两点是一致的：

- `OpenClaw + Hermes` 是否共存，不是硬性必要
- 对项目型、串行调度型工作，更重要的是隔离和顺序调度

5.2 因此，HermesGo 当前更适合继续往“低门槛单入口”推进，而不是优先扩展成多系统生态。

## 6. 归档判断

6.1 当前路线可概括为：

- 先让小白能用
- 再让社区能共建
- 最后再做评分、分发和互助网络

6.2 这是更符合现实采用路径的顺序。
