**目的**：把 HermesGo 和本仓库的开发方式从“单脚本推进”收敛为“像实际开发团队一样”的自动化流程，确保角色、过程、结果和审查都分层落盘。

## 1. 现状判断

1.1 当前仓库已经有启动器、监工、自检和文档。

1.2 真正缺的不是更多脚本，而是项目级的统一流程：

- 谁负责目标和契约
- 谁负责任务推进
- 谁负责实现
- 谁负责运行监工
- 谁负责测试
- 谁负责审查
- 谁负责归档

1.3 因此这次重构不以“再写一个窗口”为核心，而是以“建立一个团队化开发流”为核心。

## 2. 角色划分

2.1 `ArchitectureLead`

- 负责目标、边界、成功标准和契约冻结

2.2 `DeliveryManager`

- 负责拆任务、排依赖和推进阶段

2.3 `Implementer`

- 负责具体实现与本块自测

2.4 `Supervisor`

- 负责运行状态、日志、失败、超时和自动重试

2.5 `Tester`

- 负责 headless 验证和 supervisor 验证

2.6 `Reviewer`

- 负责门禁裁决，只基于证据给出结论

2.7 `Archivist`

- 负责过程日志和阶段归档

## 3. 工件目录

3.1 `workflow/`

- 保存角色定义、阶段门禁、运行脚本和运行目录

3.2 `workflow/runs/`

- 保存每次自动化开发流程的 intake、plan、contracts、implementation、verification、review、summary

3.3 `contracts/`

- 保存冻结契约

3.4 `reviews/`

- 保存审查记录

3.5 `tests/contract`、`tests/integration`、`tests/e2e`

- 保存不同层次的验证入口

## 4. 自动化流程

4.1 `workflow/Start-AutoDevFlow.ps1` 负责启动一轮开发流程。

4.2 它会自动生成：

- 立项文档
- 计划文档
- 契约文档
- 实现清单
- 验证报告
- 审查记录
- 汇总文档
- 运行日志
- 状态文件

4.3 对 HermesGo，它默认会运行两类验证：

- `Verify-HermesGo.ps1`
- `Verify-HermesGoSupervisor.ps1`

4.4 这意味着“主链路”和“监工链路”都进入了同一轮流程，而不是各自分散。

## 5. 审查门禁

5.1 没有验证输出，不得进入审查。

5.2 没有审查结论，不得算交付。

5.3 没有进度日志和运行工件，不得算过程完整。

## 6. 这次已落地的内容

6.1 已建立 `workflow/` 目录和角色/阶段文档。

6.2 已建立运行产物目录 `workflow/runs/`。

6.3 已新增 `Verify-HermesGoSupervisor.ps1`，用于补齐 supervisor 的独立验证。

6.4 已把“中大型开发任务按角色推进”写入最高规则。

## 7. 下一步

7.1 后续所有中大型任务都优先走 `workflow/Start-AutoDevFlow.ps1`。

7.2 如果 HermesGo 后面继续扩展 UI 或更多模块，应在 `contracts/` 里先冻结交接点，再并行实现。
