# AutoSkill 驱动的规则自动更新方案

**文档编号**：003

**生成者**：AI Codex GPT-5.4

**生成日期**：2026-04-15-2117

**最后修改**：2026-04-15-2117

**参考文档**：
- docs/002-2026-04-15-2039-Codex多智能体编排评估.md

**目的**：评估 AutoSkill 是否适合把对话经验转成可复用规则，并给出在本仓库里可落地的自动更新流程。

**说明**：本文把 AutoSkill 映射为“规则候选提取器 + 版本化维护器 + 人工审查门禁”，不把它当成无门槛自动改写核心规则的黑盒。

## 1. 结论

1.1 AutoSkill 可以用来做规则自动更新，但正确姿势不是直接改 `.cursorrules`，而是先自动生成“规则候选”，再经人工确认后合并。

1.2 适合自动化的对象是稳定、重复、可版本化的偏好和流程，比如写作格式、模块划分习惯、测试顺序、联调门禁。

1.3 不适合全自动的对象是一次性任务、互相冲突的偏好、带安全风险的核心约束，以及会影响全局行为的顶层规则。

1.4 如果把它用于本仓库，最现实的目标是减少重复提醒、降低上下文冗余、让规则更新有版本记录，而不是让模型自己随意改规则。

## 2. AutoSkill 机制

2.1 AutoSkill 的核心是把互动轨迹里的稳定经验抽成显式技能，再按版本维护和复用。

2.2 论文摘要明确说明，它不依赖重训底座模型，而是通过插件层把可复用技能注入后续请求。

2.3 技能生命周期可以概括为：观察、抽取、结构化、检索、复用、升级、归档。

2.4 对规则更新最有用的是“升级”步骤：将新出现的稳定偏好和旧规则比较，决定是新增、合并还是丢弃。

## 3. 在本仓库中的实施方式

3.1 先把规则分层。

3.1.1 顶层规则放 `.cursorrules`，保持少而稳。

3.1.2 项目级经验规则放独立的 `docs/` 归档中，按编号和时间版本化。

3.1.3 临时规则、一次性偏好和不稳定结论不要写进顶层规则。

3.2 再建立候选更新流程。

3.2.1 每次任务结束后，提取“本次是否出现了稳定重复的新规则”。

3.2.2 把候选规则写成结构化条目：触发条件、目标、约束、反模式、例子。

3.2.3 用一个判断步骤决定：`add`、`merge` 或 `discard`。

3.2.4 只有通过人工确认后，才允许改写 `.cursorrules` 或生成新的规则文档。

3.3 最后做版本控制。

3.3.1 每条新增规则都保留来源和日期。

3.3.2 合并规则时保留旧版本，避免规则漂移。

3.3.3 被废弃的规则不要删除历史，只标记为过期。

## 4. 推荐落地形态

4.1 最小可行形态：手工记录 + LLM 提取候选 + 人工合并。

4.2 进阶形态：加一个“规则候选评审表”，每次输出 `add/merge/discard` 和理由。

4.3 更完整形态：把规则做成独立 skill 文件，再由主规则文件引用稳定版本。

4.4 在这个仓库里，优先推荐“文档化 skill + 手工门禁”，因为它和当前 `docs/`、`docx/` 的归档习惯一致。

## 5. 预期效果

5.1 对高频重复偏好的收敛效果最好，比如固定写作风格、固定输出格式、固定测试要求。

5.2 对项目协作效率的提升来自减少重复说明，而不是让模型变得更“聪明”。

5.3 从论文信息看，AutoSkill 能从超过 2.3 万段对话中抽出 1,858 个技能，并让某些技能版本持续迭代到 `v0.1.34`，这说明它对重复场景和长期偏好有价值。

5.4 但它不是万能的：如果语料噪声大、规则互相冲突，或者缺少人工审查，效果会明显下降。

## 6. 风险与门禁

6.1 最大风险是把一次性请求误提升为长期规则。

6.2 第二个风险是 prompt injection 把错误约束写进规则库。

6.3 第三个风险是旧规则和新规则互相打架，导致行为漂移。

6.4 所以核心门禁必须保留：先提候选，再审查，再落盘，不能反过来。
