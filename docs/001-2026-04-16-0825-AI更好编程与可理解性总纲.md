## 1. 结论

1.1 AI 编程最怕的不是“不会写”，而是“边界不清、上下文过大、改动互相污染、最后没人能接手”。

1.2 所以，AI 编程必须先规划分块，再冻结契约，再逐块实现，再逐块验证。顺序反了，结果通常就会失控。

1.3 目标不是让 AI 一次性把所有东西写完，而是让 AI 在小而明确的边界内持续产出可验证成果。

1.4 如果人看不懂，AI 也很难长期稳定地改对。可理解性本身就是工程质量的一部分。

1.5 Hermes Agent、OpenClaw、OpenHarness 不在同一层，不能混为一谈。

1.6 harness 是“具体运行底座 / 框架 / 执行环境”这一层；OpenHarness 是“跨不同 harness 的统一接口或规范层”；Hermes Agent 和 OpenClaw 更像这一层上的具体产品或运行时。

1.7 从工程角度看，Hermes Agent 和 OpenClaw 更可能是“同层可替换、可迁移、可协作”的关系，而 OpenHarness 是把这些同层系统接起来的上层协议思路。这里的“协作”是通过接口和适配层实现的，不是天然同构。

## 2. 概念澄清

2.1 **harness**

2.1.1 指承载 agent 运行的具体框架、包装层或执行环境。

2.1.2 它负责把模型、工具、记忆、会话、任务执行和外部系统连起来。

2.1.3 不同 harness 往往有不同接口、任务语义和工具约定。

2.2 **Hermes Agent**

2.2.1 更像一个会记忆、会学技能、会调度任务的 agent runtime。

2.2.2 它关注的是“自己怎么干活、怎么持续演化”，而不是先定义一个跨系统统一标准。

2.2.3 在本仓库语境里，它属于“行为层 / 运行时层”。

2.3 **OpenClaw**

2.3.1 从公开资料看，它更像一个带 Gateway、助手入口或本地工作台意味的 agent 平台。

2.3.2 它与 Hermes Agent 可以被放在同一类产品层面比较，但它不是 OpenHarness 这种协议层。

2.3.3 如果后续要迁移或协作，重点在会话、入口、记忆和工具接入方式，而不是品牌名本身。

2.4 **OpenHarness**

2.4.1 它更像统一不同 harness 的开放 API / 适配规范。

2.4.2 它关心的是“不同底座如何接成一套接口”，不是“替你把任务做完”。

2.4.3 它适合做平台中台、适配层、比较层、切换层。

## 3. 关系图

3.1 更接近下面这个层次：

3.1.1 `模型 / 工具 / 会话`

3.1.2 `harness / runtime`

3.1.3 `OpenHarness 这类统一接口层`

3.1.4 `具体产品或系统：Hermes Agent、OpenClaw 等`

3.2 因此：

3.2.1 Hermes Agent 不是 OpenHarness 的替代品。

3.2.2 OpenClaw 也不是 OpenHarness 的替代品。

3.2.3 Hermes Agent 和 OpenClaw 更像可比较、可迁移、可协作的同层方案。

3.2.4 OpenHarness 更像把这些同层方案纳入同一套接入方式的上层协议思路。

## 4. AI 编程方法

4.1 **第一步：先定义目标**

4.1.1 先写清楚这次要解决什么问题，不写就容易偏题。

4.1.2 目标必须可验证，最好能说成“输入是什么、输出是什么、什么算完成”。

4.2 **第二步：先分块，再写代码**

4.2.1 一个块只做一件事。

4.2.2 每个块要有清晰输入、输出、依赖和验收标准。

4.2.3 块之间尽量用稳定契约连接，不要用临时口头约定。

4.3 **第三步：冻结契约**

4.3.1 API、数据结构、错误码、状态流先定下来。

4.3.2 契约不清，后面一定会反复返工。

4.3.3 共享层只放最小公共部分，避免把业务实现揉进去。

4.4 **第四步：逐块实现**

4.4.1 每次只让 AI 处理一个小目标。

4.4.2 每次完成后立刻验证，再进入下一块。

4.4.3 不要让 AI 同时改太多互相关联的地方。

4.5 **第五步：逐块验证**

4.5.1 先做单元级验证，再做集成验证，最后做端到端验证。

4.5.2 验证不是附属动作，是交付的一部分。

4.5.3 不能验证的改动，默认不算完成。
