## 1. 结论

1.1 **提示词工程**，主要解决“你怎么把要求说清楚”。

1.2 **上下文工程**，主要解决“模型每一步到底看见什么信息”。

1.3 **Harness 工程**，主要解决“Agent 在什么运行环境、约束系统和反馈闭环里持续干活”。

1.4 三者不是互斥替代，更像三层递进：

1.4.1 Prompt 关注一句话或一轮任务的表达。

1.4.2 Context 关注一次运行中信息如何被挑选、压缩、检索、记忆和交接。

1.4.3 Harness 关注跨会话、跨工具、跨代理、跨验证链路的整套执行系统。

1.5 如果任务只是单轮问答，Prompt 往往最重要；如果任务依赖大量外部信息，Context 更关键；如果任务要长时间自动执行并且必须稳定交付，决定成败的通常是 Harness。

## 3. 提示词工程是什么

3.1 提示词工程的核心，是把任务说明、角色、限制条件、输出格式、示例表达清楚。

3.2 它最适合的场景是：

3.2.1 单轮或短链路任务。

3.2.2 目标明确、外部依赖少的任务。

3.2.3 主要问题出在“模型没理解你的意思”。

3.3 典型做法包括：

3.3.1 明确目标和边界。

3.3.2 给 Few-shot 示例。

3.3.3 指定输出格式和检查标准。

3.3.4 把复杂任务拆成多个小提示。

3.4 它的上限也很明显：

3.4.1 当任务变成长时运行、需要不断读文件、调用工具、跨上下文接力时，光把提示词写漂亮并不够。

3.4.2 Anthropic 在 2025-09-29 的文章里已经把重点从“写一个完美提示”转向“每一步给模型最小但高信号的信息集合”。

## 4. 上下文工程是什么

4.1 上下文工程的核心，是控制模型在当前这一步看见哪些信息，不看见哪些信息，以及这些信息以什么顺序进入上下文窗口。

4.2 按 Anthropic `2025-09-29` 的总结，它关注的重点包括：

4.2.1 动态检索，而不是一次性把所有资料塞进去。

4.2.2 让 Agent 用文件路径、查询、链接等“轻引用”按需取数。

4.2.3 在长任务里做压缩、笔记、外部记忆和子代理分工。

4.3 对应的工程动作通常是：

4.3.1 RAG 或按需检索。

4.3.2 消息压缩与上下文重置。

4.3.3 `NOTES.md`、todo 文件、任务清单等持久化记忆。

4.3.4 子代理探索后只返回浓缩摘要。

4.4 这里真正优化的不是“模型智商”，而是“模型注意力预算”。

4.5 所以上下文工程比提示词工程更接近“信息编排”，而不是“措辞优化”。

## 5. Harness 工程是什么

5.1 以下定义是结合小宇宙转述与官方文章做的工程归纳：

5.1.1 Harness 工程不是单指某个 prompt、某个工具或某个框架。

5.1.2 它是把 Prompt、Context、Tools、Memory、Permissions、Workflow、Tests、Evals、Observability、Git/PR 流程、清理机制这些东西装配成一个可长期运行的系统。

5.2 OpenAI 在 `2026-02-11` 的文章里给出的信号很明确：

5.2.1 他们用 Codex 在五个月里构建并交付一个内部 beta 产品，代码、测试、CI、文档、观测和内部工具都由 Codex 写出。

5.2.2 他们把工程师角色重心，从“直接写代码”转到“设计环境、明确意图、构建反馈闭环”。

5.2.3 他们把人和 Agent 的分工压缩成一句话：`Humans steer. Agents execute.`

5.3 Anthropic 在 `2025-11-26` 与 `2026-03-24` 的两篇文章里，把 Harness 工程落得更细：

5.3.1 用 initializer agent 先铺环境、脚本、特性清单、进度文件和初始提交。

5.3.2 后续 coding agent 每次只推进一个特性，并留下结构化交接产物。

5.3.3 长任务继续升级为 planner / generator / evaluator 三代理结构，让规划、实现、验收分开。

5.4 Vercel 在 `2025-12-22` 的文章里给了一个反直觉案例：

5.4.1 他们把内部 Text-to-SQL Agent 的大部分专用工具删掉，只保留 Bash 命令执行。

5.4.2 结果是 `3.5x` 更快、`37%` 更少 token、成功率从 `80%` 到 `100%`。

5.4.3 这说明 Harness 工程不等于越复杂越好，而是要构造“模型更容易理解和操作”的环境。

5.5 所以 Harness 工程的本质，不是“再发明一个新名词”，而是把 Agent 的运行底座工程化。

## 6. 三者怎么区分

6.1 可以用一句话快速区分：

6.1.1 Prompt 工程：我怎么说。

6.1.2 Context 工程：模型此刻知道什么。

6.1.3 Harness 工程：模型在什么制度、工具、状态和反馈系统里做事。

6.2 也可以用问题来判断你当前卡在哪一层：

6.2.1 如果模型总是“误解任务”，优先查 Prompt。

6.2.2 如果模型“知道得不够、拿错资料、上下文污染”，优先查 Context。

6.2.3 如果模型“跨会话失忆、越做越乱、不会验收、不会恢复、不会交接”，优先查 Harness。

## 7. 对当前仓库语境的落点

7.1 本仓库一直在做的很多事情，其实已经接近 Harness 工程，而不只是 Prompt 工程：

7.1.1 `.cursorrules` 和 `AGENTS.md`，属于行为约束和运行规范。

7.1.2 `logs/agent-progress.md`，属于过程状态外置。

7.1.3 `docs/` 归档，属于结构化长期记忆。

7.1.4 自动验证、自检脚本、构建脚本，属于反馈闭环和可恢复机制。

7.2 也就是说，真正让 Agent 稳定工作的，往往不是“再润色一句提示词”，而是把这些外围设施补齐。

## 8. 最终判断

8.1 “提示词工程过时了”这种说法太绝对。

8.2 更准确的说法是：

8.2.1 Prompt 工程仍然有用，但它只解决最表层的问题。

8.2.2 Context 工程把问题推进到“信息供给”层。

8.2.3 Harness 工程把问题推进到“系统运行”层。

8.3 当模型越来越强时，真正拉开差距的往往不是谁提示词写得更花，而是谁给 Agent 配了更好的工作台、更清晰的约束、更干净的状态流、更可靠的验证和回退机制。
