# 033-HermesGo 大文件与发布策略归档

## 元信息
- 编号：33
- 日期：2026-04-19
- 参考：`docs/032-2026-04-19-HermesGo大文件与发布策略规划.md`、`.cursorrules`、`create_hermes_go/README.md`
- 目的：归档这次关于“大文件是否全部进 git”的决策结果，作为后续拆分仓库和发布流程的依据。

## 归档结论

本次结论是：
- 不建议把 HermesGo 的所有 zip、dll、pyd、exe、模型缓存、便携运行时全部提交到 git。
- 建议保留源码、脚本、清单、文档和必要的构建规则。
- 建议把离线运行包作为独立产物发布到网盘或 Releases。
- 建议补充下载清单和 SHA256 校验，避免“解压即用”方案失控。

## 已形成的规划

规划已经明确为四层：

1. 源码层
2. 构建层
3. 离线包层
4. 下载与校验层

其中：
- 源码层进入 git
- 构建层由脚本生成
- 离线包层外置分发
- 下载与校验层负责首次启动自动补齐

## 适合进 git 的内容

- `HermesGo/runtime/hermes-agent/hermes_cli/*.py`
- `HermesGo/runtime/hermes-agent/web/src/*`
- `HermesGo/tools/*.ps1`
- `create_hermes_go/*.ps1`
- `docs/*.md`
- 清单文件
- 校验文件

## 不适合进 git 的内容

- `runtime/python311/`
- `runtime/ollama/`
- `runtime/hermes-agent/web_dist/`
- `node_modules/`
- `*.zip`
- `*.exe`
- `*.dll`
- `*.pyd`
- `*.whl`
- 临时构建缓存

## 推荐发布方式

- 主发布：GitHub Releases
- 备用发布：百度网盘
- 验证方式：SHA256 校验
- 用户体验：下载后解压即可使用；缺大包时由脚本补齐

## 归档状态

当前建议已经可以视为定案，后续实施时优先按该规划拆分文件和发布流程。
如果后面需要进一步推进，下一步应是：
- 补 `.gitignore`
- 补资产清单
- 补首次启动下载脚本
- 把输出目录和源码目录的边界固定下来
